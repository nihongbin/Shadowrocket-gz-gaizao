# A/B/C 验证矩阵

本文件用于记录第一阶段实机验证结果。只有实机验证有效且没有明显可用性问题的配置项，才允许进入最终 v1。

## 测试分组

| 组别 | 配置文件 | 目的 | 预期用途 |
|---|---|---|---|
| A | `configs/A-johnshall-sr_cnip_ad.conf` | 原版基线 | 看 Johnshall 原版在同一环境下是否暴露 DNS/IPv6 问题 |
| B | `configs/B-nznl31-a-nomad.conf` | 外部样本 | 观察 NZNL31/a-nomad 方案的 DNS 表现和可用性代价 |
| C | `configs/C-sr_cnip_ad_privacy_hardened_candidate.conf` | 本项目候选 | 验证保留 Johnshall 主体、只增强 DNS/IPv6/回退项是否有效 |

## 固定条件

- 设备：同一台 iPhone 14。
- App：同一个 Shadowrocket 版本。
- 节点：同一个代理节点，不在三组测试中更换。
- 网络：Wi-Fi 测试时固定同一个 Wi-Fi；蜂窝测试时固定同一张 SIM 和同一位置。
- 操作：每组配置导入后，先断开 Shadowrocket，再重新连接。
- 浏览器：尽量使用同一个浏览器的无痕/隐私窗口，减少缓存影响。

## 测试步骤

1. 导入 A 组配置，选择同一个节点，断开并重新连接 Shadowrocket。
2. 记录出口 IP、DNS、IPv6、WebRTC 开关确认、国内外网站可用性。
3. 切换到 B 组配置，重复同样步骤。
4. 切换到 C 组配置，重复同样步骤。
5. 在 Wi-Fi 下完成 A/B/C 后，切到蜂窝网络，再完整重复 A/B/C。

## 测试项目

| 项目 | 测试方式 | 通过倾向 | 说明 |
|---|---|---|---|
| 出口 IP | 打开 `ipinfo.io` 或 `ip.sb` | 显示代理节点所在地 | 只用于确认节点出口一致 |
| DNS 泄露 | 打开 `dnsleaktest.com` 和 `ipleak.net` | 不明显显示本地运营商 DNS | 记录 DNS 服务商、国家/地区和数量 |
| IPv6 泄露 | 打开 `test-ipv6.com` 或 `ipv6-test.com` | 不暴露本机真实 IPv6 | 若显示本地 IPv6，记录为风险 |
| 系统 DNS 回退 | 对比 DNS 测试结果和 Shadowrocket 日志 | 不回退到本地系统 DNS | 需要结合多次测试判断 |
| 直连域名 DNS | 打开国内常用网站并观察 DNS 结果 | 不明显使用本地运营商 DNS 解析国外敏感域名 | 记录异常，不做单次绝对判断 |
| 硬编码明文 DNS | 查看 Shadowrocket 日志中是否出现 `8.8.8.8:53`、`1.1.1.1:53` 等 | C 组能拦截或不直连这些目标 | 若无法触发，标记为未验证 |
| WebRTC | 打开 `browserleaks.com/webrtc` | Shadowrocket 自带开关生效 | 不作为 C 组配置验收目标 |
| 分流可用性 | 访问国内 App/网站、国外网站、Apple 基础服务 | 无明显大面积异常 | 尤其关注银行、支付、地图、App Store |

## 采纳规则

- C 组明显优于 A 组，且没有明显可用性问题：候选项可进入最终 v1。
- B 组有效但 C 组无效：只记录 B 组有效现象，继续拆解原因，不直接复制 B 组整套策略。
- 某项无法触发或无法证明有效：写入 `docs/research-notes.md`，不进入最终结论。
- 任何配置导致国内基础服务大面积异常：不进入最终 v1，除非能找到明确、低风险修正。
